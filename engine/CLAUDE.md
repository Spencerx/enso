# engine/

The Enso **language engine** — everything needed to parse, compile, run, and
serve Enso code. Built with **SBT** from the repo root (`build.sbt` and plugins
under `project/`). Written in Scala + Java (with a nightly Frgaal Java compiler
so we can use modern Java features across the matrix). Runs on **GraalVM** using
the **Truffle** framework.

## Top-level modules

- `runtime/` — The Truffle-based interpreter. Parser Java bindings + JNI native
  library live here. Contains the IR, compiler passes, the Truffle AST, builtin
  methods, and the runtime-server instrument that exposes live execution info.
- `language-server/` — Enso Language Server. Talks JSON-RPC + binary data-link
  over WebSocket to IDEs (our Electron client, VS Code, etc.).
- `runner/` — CLI entry point (`enso` / `ensoup` command). Parses args,
  constructs a GraalVM polyglot `Context`, loads the runtime language.
- `polyglot-api/` + `polyglot-api-macros/` — Stable wire-format types used to
  cross the language/embedder boundary between `runtime/` (Truffle-side) and
  `language-server/` (embedder-side). Uses `enso-persistance` for custom
  serialization.
- `common/` — Shared constants/interfaces used by multiple engine submodules.
- `launcher/` — The `enso` launcher binary (Native Image-compiled) that resolves
  the right engine version and invokes it.
- `runner-common/` — Shared between multiple runner variants (CLI, Jupyter
  kernel, REPL).
- `runtime-parser/`, `runtime-parser-dsl/`, `runtime-parser-processor/`,
  `runtime-parser-processor-tests/` — Java bindings to the Rust parser and
  annotation-processor pipeline that builds them.
- `runtime-compiler/`, `runtime-compiler-dump/`, `runtime-compiler-dump-igv/` —
  Compiler (IR + passes) extracted as its own module for native-image
  friendliness. IGV support = integration with the Graal visualizer (see
  `tools/enso4igv/`).
- `runtime-instrument-*` — Truffle instruments. `common` is shared
  infrastructure; the others are the REPL debugger, IDE runtime-server, and
  expression-id instrumentation.
- `runtime-integration-tests/` — Scalatests that drive the whole language
  end-to-end. Our primary test surface for the engine.
- `runtime-benchmarks/`, `runtime-test-instruments/` — JMH benchmarks +
  test-only instruments.
- `runtime-language-arrow/`, `runtime-language-epb/` — The **Enso Polyglot
  Bridge (EPB)** — an internal sub-language that wraps single-threaded Truffle
  languages (JS, R) so the multi-threaded Enso language can call them safely.
  Arrow = columnar data interop.
- `runtime-suggestions/` — Code-completion/suggestion database maintained by the
  runtime and read by the LS.
- `interpreter-dsl-test/` — Tests for the annotation-processor DSL used to
  generate Truffle nodes.
- `python-resource-provider/` — Native helper that exposes Python venv resources
  to the runtime.

## Key engineering conventions

(See `docs/runtime-guide.md` for the long version, and
`docs/style-guide/{java,scala}.md`.)

- **Nodes are built, not new'd.** Every Truffle `Node` in an AST has a
  `public static build(...)` method. DSL-generated nodes delegate `build` to the
  autogenerated `create`. This lets us flip a node between DSL and hand-written
  without disturbing callers. See `engine/runtime/README.md`.
- **Two APIs, don't mix.** Language-side code uses `InteropLibrary`;
  embedder-side code uses `org.graalvm.polyglot.Value`. Only the runner is
  embedder-side. If `org.graalvm.polyglot` appears in a `runtime/` file, that's
  a bug.
- **Safepoints and threading.** Enso has a custom thread manager
  (`ThreadManager`) that polls safepoints at method boundaries and TCO
  iterations. Don't block the Truffle thread with arbitrary Java code without
  yielding to safepoints.
- **Builtins.** The `Standard.Builtins` virtual module is synthesized from
  `runtime/runtime/builtin/*`. A Java-side annotation DSL generates boilerplate
  (`@BuiltinMethod`, `@Specialization`). See
  `docs/runtime/builtin-base-methods.md`.
- **Entry points.** Embedder-side: `Main` in `engine-runner`. Language-side:
  `EnsoLanguage` in `runtime/`. IDE flow goes via `language-server` →
  `RuntimeServerInstrument` → `runtime/`. They exchange messages via
  polyglot-api and stringly-typed method names, so chase constants to cross the
  boundary.

## Testing

- `sbt runtime-integration-tests/test` — language semantics + compiler passes.
- `sbt runtime-benchmarks/bench` — microbenchmarks. Appends to
  `bench-report.xml`.
- Per-library tests in `test/` (Enso code) run via
  `sbt 'runEngineDistribution --run test/…'`.
- See `docs/runtime-guide.md` §"Code & Internal Documentation Map" for the map
  of entry points.

## Gotchas

- Building the runtime is slow (native-image stages + annotation processing).
  The first full build can take 10+ minutes. Incremental SBT builds are much
  faster.
- The Scala/Java source split is the result of an **ongoing** migration from
  Scala to Java (see `docs/runtime-roadmap.md`). New code should lean Java-first
  unless it benefits from Scala's type system; mixing is fine but inheritance
  across languages needs care.
- `language-server` and `runtime` talk via binary Ydoc frames (via the polyglot
  `ydoc-server-polyglot`) when ydoc is enabled. The polyglot variant runs
  _inside_ the engine's GraalJS — not in Node.
